# AI Test Agent TypeScript 重构项目功能规格说明书

## 一、引言

### 1.1 目的

本文档作为 AI Test Agent TypeScript 重构项目的功能规格说明书，旨在详细描述每个功能需求的实现方案、设计决策和技术实现细节。本文档是架构设计文档的具体化，为开发团队提供明确的实现指导，是代码编写和单元测试的直接依据。

本文档与需求规格说明书形成对应关系，每个功能需求章节都与需求规格说明书中的相应章节对应，确保所有需求都有对应的实现方案。对于需求规格说明书中未明确的技术决策，本文档进行补充和细化。

### 1.2 范围

本功能规格说明书覆盖系统的全部核心功能模块，包括驱动模块、自愈模块、MCP 协议模块、Agent 智能体模块、技能模块和知识库模块。每个模块按照组件描述、数据结构、接口说明、算法流程四个维度进行详细描述。

本说明书不包括具体的代码实现，实现细节由开发人员在遵循本说明书的前提下自行决定。不包括 UI 界面设计，命令行接口的设计在此说明但不涉及图形界面。不包括性能优化方案，具体优化措施在实现过程中根据测试结果决定。

功能规格说明书中使用的专业术语定义如下：

组件是指实现特定功能的代码单元，具有明确的接口和内部实现。接口是组件对外暴露的操作集合，定义了组件的交互方式。数据结构是组件用于存储和传递数据的类型定义。算法流程是组件完成特定功能的处理步骤和逻辑顺序。异常处理是组件对错误情况的响应机制和处理方式。

## 二、驱动模块功能规格

### 2.1 模块概述

驱动模块封装了与 HarmonyOS 设备交互的所有底层能力，是整个系统的操作基础。模块采用工厂模式创建驱动实例，支持多种配置方式。模块内部划分为五个子模块：驱动核心子模块负责连接管理和命令转发，选择器子模块负责元素定位策略实现，操作封装子模块负责高级操作序列的封装，断言库子模块负责验证逻辑的实现，配置管理子模块负责运行时配置的处理。

### 2.2 驱动核心子模块

#### 2.2.1 HypiumDriver 主类

HypiumDriver 主类采用单例模式管理设备连接，确保同一时间只有一个活跃的设备连接。主类接收 DriverConfig 配置对象作为构造参数，包括设备地址、端口号、默认超时时间、重试次数、截图保存目录和日志配置等。

主类提供以下核心方法：connect 方法异步建立与设备的 TCP 连接，连接成功后启动心跳保活机制，每 30 秒发送一次心跳包检测连接状态；disconnect 方法关闭连接并停止心跳，释放所有资源；isConnected 方法返回当前连接状态布尔值。

主类内部维护设备状态机，状态包括 DISCONNECTED（未连接）、CONNECTING（连接中）、CONNECTED（已连接）和 ERROR（错误）。状态转换通过事件触发，状态变化时发出相应事件通知观察者。

#### 2.2.2 设备命令协议

设备命令采用基于 HDC 工具的自定义协议封装。命令格式为 JSON 对象，包含 command（命令名称）、params（参数字典）和 requestId（请求标识）三个字段。响应格式同样为 JSON 对象，包含 requestId、status（状态码）和 result（结果数据）三个字段。

状态码采用 HTTP 风格设计：200 表示成功，400 表示请求参数错误，404 表示元素未找到，500 表示服务器内部错误，504 表示执行超时。每种错误状态都包含详细的错误信息，便于问题诊断。

命令执行采用 Promise 封装，支持链式调用。命令队列机制确保命令按顺序执行，避免并发冲突。对于需要等待响应的命令，系统设置默认 30 秒超时，超时后自动重试指定次数。

#### 2.2.3 连接池管理

为了提高连接效率，系统实现了连接池管理机制。连接池维护固定数量的活跃连接，根据负载动态调整。连接池支持以下配置参数：minConnections 表示最小连接数，默认 1；maxConnections 表示最大连接数，默认 5；acquireTimeout 表示获取连接超时时间，默认 10 秒；idleTimeout 表示空闲连接存活时间，默认 5 分钟。

连接池采用懒加载策略，只有在实际需要时才创建新连接。空闲连接在超过存活时间后被自动关闭，释放系统资源。连接池提供连接健康检查功能，定期检测空闲连接的可用性。

### 2.3 选择器子模块

#### 2.3.1 选择器架构

选择器子模块采用策略模式实现多种定位策略。每种策略实现统一的 Selector 接口，包含 locate 和 locateAll 两个方法。locate 方法返回第一个匹配的元素，locateAll 方法返回所有匹配的元素列表。

选择器管理器维护策略优先级和回退链条。默认优先级顺序为：ID 选择器、文本选择器、XPath 选择器、坐标选择器和视觉选择器。当高优先级策略定位失败时，选择器管理器自动尝试下一级策略，直到找到元素或所有策略都失败。

选择器支持组合使用，可以通过 AND、OR、NOT 等逻辑运算符组合多个定位条件。组合选择器首先计算每个子选择器的结果，然后进行逻辑运算得到最终结果。

#### 2.3.2 文本选择器实现

文本选择器通过 OCR 技术识别屏幕上的文字元素。实现步骤包括：首先调用截图接口获取当前屏幕图像；然后使用 Tesseract.js 或系统原生 OCR API 识别图像中的文字；接着将识别的文字与目标文本进行匹配；最后返回包含目标文本的元素信息。

文本匹配支持精确匹配和模糊匹配两种模式。精确匹配要求文字完全一致，模糊匹配允许部分匹配并根据相似度阈值过滤结果。模糊匹配的相似度计算采用编辑距离算法，阈值为 0.8，即匹配度超过 80% 的结果被视为有效。

文本选择器支持通配符匹配，使用正则表达式语法。例如，“登录.*” 可以匹配“登录按钮”、“登录页面”等文字。

#### 2.3.3 ID 选择器实现

ID 选择器使用 HarmonyOS 元素的 resource-id 属性进行定位。每个元素在 HarmonyOS 应用中都有唯一的资源标识符，ID 选择器通过精确匹配 resource-id 来找到目标元素。

ID 选择器首先获取当前界面的 UI 树，然后遍历所有元素匹配 resource-id 属性。匹配成功的第一个元素作为定位结果返回。ID 选择器是最可靠和最快的定位方式，优先推荐使用。

ID 选择器支持部分匹配模式，当 resource-id 包含指定字符串时即视为匹配。部分匹配使用 endsWith、startsWith 和 contains 三种模式，通过配置参数指定。

#### 2.3.4 XPath 选择器实现

XPath 选择器支持 W3C XPath 规范的子集，实现以下轴定位：child 表示子元素，parent 表示父元素，descendant 表示后代元素，ancestor 表示祖先元素，following-sibling 表示后续兄弟元素，preceding-sibling 表示前置兄弟元素。

XPath 选择器支持的函数包括：text 用于文本匹配，contains 用于部分匹配，matches 用于正则匹配，count 用于统计数量，position 用于位置选择。

XPath 选择器的实现首先解析 XPath 表达式为抽象语法树，然后根据语法树在 UI 树上进行递归遍历，最后收集满足条件的元素作为结果。

#### 2.3.5 视觉选择器实现

视觉选择器利用 AI 模型识别目标元素。实现原理是：首先使用目标元素的截图作为模板，然后在屏幕截图上使用图像识别算法搜索匹配区域，最后将匹配区域的中心坐标作为元素位置返回。

视觉选择器支持多模板匹配，可以同时使用多个模板进行搜索，提高识别准确率。模板匹配使用 SIFT 或 SURF 算法实现，具有旋转和尺度不变性。

视觉选择器支持自定义阈值配置，阈值范围为 0 到 1，值越高要求匹配越精确。默认阈值为 0.85，低于此值的匹配结果被忽略。

### 2.4 操作封装子模块

#### 2.4.1 HypiumActions 主类

HypiumActions 主类封装了常用的操作序列，提供语义化的操作接口。主类依赖 HypiumDriver 实例进行底层操作，通过组合模式提供高级功能。

主类提供以下操作方法：navigateTo 方法接收页面名称参数，自动执行导航到目标页面的操作序列；fillForm 方法接收键值对字典，自动填写表单字段；executeChain 方法接收操作链对象，顺序执行一系列操作。

操作方法都支持延迟参数，指定操作执行前等待的时间。延迟参数使用毫秒为单位，默认等待 500 毫秒。操作完成后也可以指定后延迟，用于等待界面响应。

#### 2.4.2 ActionChain 实现

ActionChain 是操作链的数据结构，支持链式调用语法。链对象包含操作列表，每个操作包含类型、目标和参数三个属性。类型枚举包括 tap、input、swipe、wait、screenshot 等。

ActionChain 的 execute 方法顺序执行操作列表中的每个操作。执行过程中遇到错误时，根据配置决定是中断执行还是跳过当前操作继续执行下一个。执行完成后返回详细的执行报告，包含每个操作的开始时间、结束时间和执行结果。

操作链支持嵌套，可以在操作序列中嵌入子操作链。子操作链作为独立单元执行，执行完成后返回子操作链的结果。

### 2.5 断言库子模块

#### 2.5.1 HypiumAssertions 主类

HypiumAssertions 主类提供断言验证功能。主类包含软断言集合和硬断言集合两个内部状态。软断言集合收集所有验证失败但不中断执行，硬断言集合在首次失败时立即中断。

主类提供以下断言方法：assertTextExists 验证指定文本是否出现在屏幕上；assertComponentVisible 验证指定元素是否处于可见状态；assertComponentEnabled 验证指定元素是否处于可用状态；assertTextEquals 验证指定元素的文本内容是否与预期相等。

断言方法返回 AssertionResult 对象，包含 passed（是否通过）、expected（期望值）、actual（实际值）和 message（描述信息）四个属性。软断言的结果通过 evaluate 方法统一获取。

#### 2.5.2 断言重试机制

对于不稳定的断言条件，系统实现了断言重试机制。重试配置包括最大重试次数、重试间隔和重试条件。默认配置为最多重试 3 次，每次重试间隔 1 秒。

断言重试支持条件指定，只有满足条件的断言失败才进行重试。例如，可以配置只有超进出场动画导致的断言失败才重试，而元素完全不存在时不重试。

重试过程中每次尝试的结果都被记录，最终返回所有尝试的综合结果。如果多次尝试中有任意一次通过，最终结果视为通过。

### 2.6 配置管理子模块

#### 2.6.1 配置结构

配置管理子模块定义了完整的配置类型层级。顶层配置包含 server（服务器配置）、driver（驱动配置）、agent（智能体配置）、logging（日志配置）和 mcp（MCP 配置）五个子配置块。

驱动配置包含 host（设备地址）、port（端口号）、timeout（超时时间）、retryCount（重试次数）、screenshotDir（截图目录）、enableLogging（启用日志）和 connectionPool（连接池配置）等参数。

配置支持环境变量覆盖，环境变量名称采用大写下划线格式，如 DRIVER_HOST 覆盖 driver.host 配置。环境变量优先于配置文件生效。

#### 2.6.2 配置验证

配置加载时进行全面的验证检查。必填配置项必须在配置中存在且非空，否则抛出 ValidationError 异常。可选配置项使用默认值填充。配置类型必须与定义的类型匹配，不匹配时尝试类型转换，转换失败时抛出异常。

配置值范围验证对于有范围限制的配置项生效。例如，超时时间必须为正数且不超过 300 秒，屏幕截图质量必须在 0 到 100 之间。

## 三、自愈模块功能规格

### 3.1 模块概述

自愈模块实现了测试失败的自动诊断和修复能力，是系统智能化水平的核心体现。模块采用管道模式组织工作流程，数据在诊断器、决策引擎、执行器和策略之间按顺序传递和处理。

模块设计了完善的事件通知机制，每个处理阶段都可以发出事件，供外部系统监控和集成。模块支持插件化扩展，可以通过注册新的策略来增加修复能力。

### 3.2 诊断器子模块

#### 3.2.1 FailureDiagnoser 主类

FailureDiagnoser 主类是诊断流程的入口点。主类接收错误对象和诊断上下文作为输入，输出诊断结果。诊断上下文字典包含错误名称、错误消息、错误堆栈、定位策略、目标元素和历史尝试次数等信息。

主类的 diagnose 方法执行以下步骤：首先对错误进行分类，确定失败类型；然后计算诊断置信度；接着提取诊断证据；最后构建并返回诊断结果对象。

诊断结果包含 failureType（失败类型）、confidence（置信度，0-100）、rootCause（根本原因）、detailedCause（详细原因）、context（诊断上下文）、suggestedStrategies（建议策略列表）和 evidence（证据列表）七个属性。

#### 3.2.2 ErrorClassifier 错误分类器

ErrorClassifier 错误分类器维护一套基于正则表达式的分类规则。每条规则包含模式（正则表达式）、失败类型和权重三个属性。分类时遍历所有规则，找到第一个匹配的模式，返回对应的失败类型。

预定义的分类规则覆盖了 10 种标准失败类型：ELEMENT_NOT_FOUND 匹配 not found、cannot find、unable to locate 等模式；ELEMENT_NOT_VISIBLE 匹配 not visible、is not displayed 等模式；AUTH_EXPIRED 匹配 token.*expire、auth.*expire、session.*expire、401、403 等模式；ASSERTION_TOO_STRICT 匹配 assert.*fail、expected.*but got 等模式。

权重表示该规则匹配时的置信度系数，范围为 0 到 1。置信度由规则权重和上下文信息综合计算得出。

#### 3.2.3 证据收集

诊断过程收集多种证据用于后续分析和学习。错误证据包含错误名称、消息和堆栈跟踪的文本内容。截图证据是在错误发生时立即截取的屏幕图像。UI 树证据是错误发生时的完整界面元素树快照。日志证据是相关时间范围内的执行日志片段。

证据按时间戳排序存储，便于后续按时间线回放诊断过程。证据支持压缩存储，大型证据如截图和 UI 树进行 gzip 压缩后存储。

### 3.3 决策引擎子模块

#### 3.3.1 FixDecisionEngine 主类

FixDecisionEngine 主类根据诊断结果决定使用哪种修复策略。主类维护策略优先级表和历史成功率统计。决策过程综合考虑诊断置信度、历史成功率和剩余尝试次数。

主类的 decide 方法执行以下步骤：首先根据失败类型筛选适用的策略；然后按优先级和历史成功率排序策略；接着检查策略是否还有尝试机会；最后返回优先级最高的策略作为决策结果。

决策过程记录完整的决策日志，包含输入参数、排序依据和最终选择。决策日志支持导出，用于分析决策质量和优化策略排序。

#### 3.3.2 置信度计算

置信度计算采用加权评分模型。总评分由诊断置信度、历史成功率和上下文匹配度三个维度加权求和得出。诊断置信度权重为 0.5，历史成功率权重为 0.3，上下文匹配度权重为 0.2。

历史成功率按修复策略类型统计。系统维护每个策略类型的历史修复记录，包含成功次数、失败次数和最后一次执行时间。统计数据按时间衰减，最近的记录权重更高。

上下文匹配度计算当前诊断上下文与历史修复上下文的相似度。使用文本相似度算法计算上下文字符串的匹配程度。

### 3.4 执行器子模块

#### 3.4.1 FixExecutor 主类

FixExecutor 主类协调整个修复流程的执行。主类接收诊断器、决策引擎和策略列表作为构造参数。主类维护修复历史记录，用于策略选择和学习。

主类的 fixAndRetry 方法是核心执行入口。方法接收测试用例、差异结果、驱动实例和上下文作为参数，返回修复执行结果。执行流程包含诊断、决策、生成补丁、应用补丁和验证结果五个阶段。

修复流程支持中断和恢复。执行过程中可以暂停修复，保存当前状态，待条件满足后继续执行。状态包含当前阶段、已尝试的策略和执行的补丁列表。

#### 3.4.2 重试策略

修复流程采用智能重试策略。重试触发条件是修复应用后验证结果仍然失败。重试时更换为次优策略，避免重复尝试相同策略浪费时间。

重试间隔采用指数退避算法。首次重试间隔 1 秒，第二次 2 秒，第三次 4 秒，以此类推。最大间隔封顶为 30 秒，避免无限等待。

重试过程记录每次尝试的详细信息，包括策略类型、执行时间、执行结果和失败原因。记录支持查询，用于分析修复失败的原因和优化策略。

### 3.5 策略子模块

#### 3.5.1 FixStrategy 基类

FixStrategy 基类定义了所有修复策略的通用接口。基类包含 type（策略类型）、canHandle（是否能处理指定失败类型）、generateFix（生成修复补丁）和 applyFix（应用修复补丁）四个抽象方法。

generateFix 方法接收诊断结果和驱动实例，返回 FixPatch 修复补丁对象。FixPatch 包含 id（补丁标识）、strategy（策略类型）、changes（变更内容）、confidence（置信度）和 description（描述）五个属性。

applyFix 方法接收补丁对象和驱动实例，执行实际的修复操作。应用前验证补丁格式是否正确，应用后验证是否达到预期效果。

#### 3.5.2 SelectorFixStrategy 选择器修复策略

SelectorFixStrategy 实现选择器修复功能。当元素定位失败时，该策略尝试切换到其他定位方法。

策略实现了三级回退链条。第一级回退从文本定位切换到 ID 定位，切换依据是元素可能存在 resource-id 属性。第二级回退从 ID 定位切换到 XPath 定位，通过分析元素层级关系构建 XPath 表达式。第三级回退从 XPath 定位切换到视觉定位，使用元素截图作为模板进行图像匹配。

每次回退尝试前先检查目标元素是否具有相应的属性用于新定位方法。例如，切换到 ID 定位前先检查 UI 树中是否存在包含 resource-id 的候选元素。

#### 3.5.3 AuthFixStrategy 认证修复策略

AuthFixStrategy 实现认证修复功能。当检测到认证过期或失败时，该策略自动执行重新认证操作。

策略支持两种修复方式。重新登录方式需要用户提供登录凭据，执行完整的登录流程包括输入用户名密码、点击登录按钮和验证登录结果。Token 刷新方式调用 OAuth 刷新接口获取新的访问令牌，无需用户重新登录。

策略执行前先判断认证状态，确定使用哪种修复方式。判断依据是错误消息中是否包含 expired 或 invalid 等关键词，以及是否有可用的刷新令牌。

#### 3.5.4 AssertionFixStrategy 断言修复策略

AssertionFixStrategy 实现断言修复功能。当断言失败但可能存在误判时，该策略自动调整断言条件。

策略支持两种调整方式。放宽匹配方式将精确匹配改为模糊匹配，降低匹配阈值或使用包含匹配代替完全匹配。降低阈值方式对于数值比较类断言，将预期值的精度要求降低，允许一定范围内的偏差。

策略执行前先分析断言失败的具体原因，区分是测试用例错误还是被测系统问题。只有在确定是测试用例过于严格时才执行断言修复，避免掩盖真实的系统缺陷。

#### 3.5.5 DataFixStrategy 数据修复策略

DataFixStrategy 实现数据修复功能。当检测到测试数据问题时，该策略执行数据修复操作。

策略支持三种修复方式。数据清理方式清除应用缓存和临时文件，释放被占用的存储空间。数据重置方式将应用数据恢复到初始状态，适用于污染场景。账号切换方式在多账号测试场景下切换到备用账号继续测试。

策略执行前先保存当前数据状态，便于修复失败时回滚。执行过程中记录详细的操作日志，包括清理了哪些文件、重置了哪些数据。

#### 3.5.6 WaitFixStrategy 等待修复策略

WaitFixStrategy 实现等待修复功能。当操作因超时失败时，该策略调整等待策略。

策略支持四种等待方式。梯度等待方式按倍数增加等待时间，首次失败等待 1.5 倍时间，第二次 2.25 倍，第三次 3.375 倍。显式等待方式使用轮询机制持续检查目标条件，每 500 毫秒检查一次。空闲等待方式等待系统进入空闲状态后再执行操作。分步等待方式将一个复杂操作分解为多个简单步骤，逐步执行。

策略执行前先分析超时原因，判断是网络延迟、界面渲染还是服务器响应慢。针对不同原因选择最合适的等待方式。

## 四、MCP 协议模块功能规格

### 4.1 模块概述

MCP 协议模块实现了 Model Context Protocol 的 JSON-RPC 2.0 版本，提供了标准化的工具调用接口。模块分为协议层、服务端和客户端三个部分，各自独立又相互协作。

协议层实现了 JSON-RPC 消息的编解码和验证逻辑。服务端实现了工具注册、请求路由和调用分发。客户端实现了连接管理、工具发现和调用封装。

### 4.2 协议层子模块

#### 4.2.1 JSON-RPC 协议实现

JSON-RPC 协议实现严格遵循 JSON-RPC 2.0 规范。协议类提供四个核心方法：createRequest 创建请求消息，createResponse 创建成功响应，createErrorResponse 创建错误响应，createNotification 创建通知消息。

请求消息格式为 JSON 对象，字段包括 jsonrpc（固定值 "2.0"）、id（请求标识，可为字符串或数字）和 method（方法名）。params 字段为可选项，用于传递方法参数。

响应消息格式同样为 JSON 对象，字段包括 jsonrpc（固定值 "2.0"）、id（关联的请求标识）和 result（结果数据）或 error（错误对象）。result 和 error 互斥，不能同时出现。

错误对象包含 code（错误码）、message（错误消息）和 data（附加数据，可选）三个字段。JSON-RPC 2.0 定义了 -32700 到 -32600 的标准错误码，系统在 -32000 到 -32099 范围内定义业务错误码。

#### 4.2.2 消息验证

消息验证确保接收到的消息符合 JSON-RPC 2.0 规范。验证使用 Zod Schema 进行，Schema 定义了所有字段的类型约束和必填要求。

请求消息验证检查 jsonrpc 版本号是否为 "2.0"，method 是否为非空字符串，id 是否为字符串或数字类型。响应消息验证检查是否包含 result 或 error 字段，id 是否与请求匹配。

验证失败时抛出 JsonRpcError 异常，异常包含错误码和详细描述。异常被捕获后转换为格式化的错误响应。

### 4.3 服务端子模块

#### 4.3.1 MCPServer 主类

MCPServer 主类是 MCP 服务端的核心组件，负责接收请求、管理工具和处理响应。主类维护工具注册表和请求处理器列表。

主类提供工具注册方法 registerTool，接收工具定义和处理器函数作为参数。工具定义使用 ToolDefinition 结构描述，包含 name（名称）、description（描述）、inputSchema（输入模式）和 outputSchema（输出模式）。处理器函数接收参数字典，返回 Promise<ToolResult>。

主类的 handleRequest 方法是请求处理的入口点。方法接收 JSON 字符串作为输入，解析后根据方法名路由到对应的处理器，最后返回格式化的响应字符串。

#### 4.3.2 工具注册表

工具注册表维护所有已注册工具的元信息和处理器映射。注册表使用 Map 数据结构存储，键为工具名称，值为包含定义和处理的二元组。

注册表支持动态更新，可以在运行时注册新工具或注销已有工具。注册表提供查询方法 listTools，返回所有可用工具的简要信息列表。

工具注册表支持命名空间隔离。带命名空间的工具名称格式为 namespace:toolname，如 harmonyos:click 和 assert:text_exists。命名空间便于工具的组织和管理。

#### 4.3.3 请求路由器

请求路由器根据请求中的方法名将请求分发给对应的处理器。路由器维护一个方法名到处理器函数的映射表，对于带命名空间的方法名，先提取命名空间前缀，再查找对应处理器。

路由器支持方法别名功能，可以为同一处理器注册多个方法名。别名在注册时指定，如将 click 和 harmonyos_click 都映射到同一个点击处理器。

路由器在分发前执行参数验证，根据工具定义中的 inputSchema 校验参数类型和必填性。验证失败返回错误响应，不调用处理器。

### 4.4 客户端子模块

#### 4.4.1 MCPClient 主类

MCPClient 主类是 MCP 客户端的核心组件，负责建立连接、发现工具和发起调用。主类维护服务器连接和已发现工具列表。

主类的 connect 方法接收请求处理器函数作为参数，用于发送请求并接收响应。方法内部初始化连接状态，建立与服务器的通信通道。

主类的 initialize 方法执行 MCP 协议初始化流程。流程包括发送 initialize 请求、接收服务器能力声明和调用 tools/list 获取可用工具列表。初始化完成后客户端进入就绪状态，可以发起工具调用。

#### 4.4.2 工具发现

工具发现是客户端了解服务器能力的重要步骤。初始化时客户端调用 tools/list 方法获取所有可用工具的定义列表。工具列表缓存在客户端本地，减少后续调用的网络开销。

工具列表支持增量更新，客户端可以调用 tools/list_refresh 仅获取新增或变更的工具。对于长时间运行的客户端，定期刷新工具列表可以发现服务器的功能更新。

工具信息包含完整的方法签名，供 LLM 选择合适的工具使用。工具描述支持多语言，便于在不同语言环境下展示。

#### 4.4.3 工具调用

工具调用通过 callTool 方法实现。方法接收工具名称和参数字典作为输入，返回 Promise<ToolResult>。调用前验证参数格式是否符合工具定义，拒绝不符合的调用。

调用支持超时控制，默认超时时间为 60 秒。对于长时间运行的工具如批量截图，可以传入自定义超时参数覆盖默认值。

调用失败时根据错误类型进行不同处理。协议错误如超时或网络断开时自动重试，重试次数可配置。业务错误如参数错误或工具内部错误直接返回，不进行重试。

### 4.5 工具定义

#### 4.5.1 HarmonyOS 设备工具

系统预定义了 9 种 HarmonyOS 设备操作工具。harmonyos_click 点击工具接收 locator_type 和 locator_value 两个参数，执行点击操作。harmonyos_swipe 滑动工具接收 direction 和 distance 两个参数，执行滑动操作。harmonyos_input_text 输入文本工具接收 target_locator 和 text 两个参数，执行文本输入操作。harmonyos_back 返回键工具无参数，执行返回操作。harmonyos_home Home键工具无参数，执行返回主页操作。harmonyos_screenshot 截图工具接收 file_path 参数，保存截图到指定路径。harmonyos_get_ui_tree UI树工具无参数，获取当前界面的元素树。harmonyos_press_key 按键工具接收 key_name 参数，执行指定按键操作。harmonyos_long_press 长按工具接收 locator 和 duration 参数，执行长按操作。

#### 4.5.2 断言工具

系统预定义了 3 种断言验证工具。assert_text_exists 文本存在断言接收 text 参数，验证指定文本是否出现在屏幕上。assert_component_visible 组件可见断言接收 locator 参数，验证指定元素是否可见。assert_toast Toast检查工具接收 message 参数，验证是否出现包含指定文本的 Toast 消息。

断言工具的返回结果包含 passed（是否通过）和 message（描述信息）两个字段。断言失败时 passed 为 false，message 包含期望值和实际值的对比信息。

#### 4.5.3 辅助工具

系统预定义了 2 种辅助工具。ai_vision_recognize 视觉识别工具接收 target_image 参数，使用 AI 模型识别目标图像在屏幕上的位置。screenshot_compare 截图对比工具接收 expected_image 和 threshold 参数，比较实际截图与预期截图的差异度。

辅助工具用于复杂测试场景。视觉识别工具可以定位不规则形状或动态内容的元素。截图对比工具用于验证界面是否发生意外的视觉变化。

## 五、Agent 智能体模块功能规格

### 5.1 模块概述

Agent 智能体模块是整个系统的核心决策单元，实现了 ReAct 循环模式。模块整合了 LLM 决策能力、MCP 调用能力和 Fixer 自愈能力，能够自主完成测试任务。

模块采用事件驱动的设计，所有关键操作都发出事件，便于外部系统监控和集成。模块支持状态管理，能够在中断后从断点恢复执行。

### 5.2 ReAct 循环子模块

#### 5.2.1 ThinkStep 思考步骤

ThinkStep 实现 ReAct 循环的思考阶段。步骤接收当前任务描述、历史执行记录和可用工具列表作为输入，输出 LLM 决策结果。

步骤构建包含丰富上下文的提示词。提示词包含任务目标描述、历史步骤及其结果、可用工具及其使用说明、以及决策要求。提示词模板支持自定义，便于针对不同场景优化。

步骤调用 LLM 提供者的推理接口获取决策。接口调用使用函数调用格式，支持工具选择和参数生成的联合输出。接口调用支持超时控制和错误重试。

#### 5.2.2 ActStep 行动步骤

ActStep 实现 ReAct 循环的行动阶段。步骤接收工具名称和参数作为输入，通过 MCP 客户端执行工具调用。

步骤首先验证工具是否存在，不存在时抛出 ToolNotFoundError。步骤然后调用 MCPClient 的 callTool 方法发起调用。步骤最后捕获调用结果，包括成功响应和错误异常。

步骤支持重试机制。对于可重试的错误如网络超时，步骤自动按配置重试。对于不可重试的错误如参数错误，步骤直接抛出异常。

#### 5.2.3 ObserveStep 观察步骤

ObserveStep 实现 ReAct 循环的观察阶段。步骤接收任务描述、思考结果和行动结果作为输入，输出步骤是否通过的判断。

步骤同样使用 LLM 进行判断。提示词包含任务目标、执行的思考过程、工具调用的结果和当前界面状态。提示词要求 LLM 判断执行结果是否满足任务目标。

观察判断支持自动修复建议。当判断为不通过时，LLM 额外输出可能的修复建议，供后续 Fixer 流程参考。

### 5.3 LLM 提供者子模块

#### 5.3.1 LLMProvider 基类

LLMProvider 基类定义了 LLM 提供者的标准接口。基类包含推理方法 think、嵌入方法 embed 和模型信息方法 getModelInfo。

think 方法接收提示词和选项参数，返回 LLM 的推理结果。结果包含文本输出和可能的函数调用。选项包含 temperature、maxTokens 等生成参数。

基类定义了流式输出支持。streamThink 方法返回异步生成器，yield 每个 token 用于实时显示。

#### 5.3.2 OpenAI 实现

OpenAI 实现使用 OpenAI API 作为推理后端。实现封装了 OpenAI SDK，处理 API 认证、请求构建和响应解析。

实现支持 GPT-4 和 GPT-3.5-Turbo 模型，通过配置指定使用的模型。实现处理函数调用格式的转换，将 OpenAI 的 function_call 格式转换为系统内部的函数调用结构。

实现支持代理配置和自定义 API 地址，便于在企业内网环境使用。

#### 5.3.3 Qwen 实现

Qwen 实现使用阿里云通义千问 API 作为推理后端。实现同样封装了通义千问 SDK，处理阿里云的认证机制和 API 调用。

实现支持 qwen-turbo、qwen-plus 和 qwen-max 等模型，按性能和成本平衡选择。

实现处理通义千问特有的 function call 格式转换。

### 5.4 上下文管理子模块

#### 5.4.1 ExecutionContext 执行上下文

ExecutionContext 保存单个测试用例执行过程中的所有状态。上下文包含 testCase（关联的测试用例）、currentStepIndex（当前步骤索引）、steps（步骤历史列表）、variables（变量映射）、startTime（开始时间）和 endTime（结束时间）等属性。

上下文支持状态快照功能。快照包含完整的状态信息，可以序列化后存储到磁盘。后续可以从快照恢复执行，适用于长时间任务的断点续传。

上下文支持变量绑定。步骤执行结果可以绑定到变量，供后续步骤引用。变量使用 ${variableName} 语法在步骤描述中引用。

#### 5.4.2 HistoryTracker 历史追踪

HistoryTracker 维护完整的执行历史记录。记录包含每个步骤的思考内容、选择的工具、传入的参数、返回的结果和最终判断。

历史记录用于分析和优化。提供了统计方法计算平均步骤数、平均执行时间、成功率等指标。也提供了查询方法按条件筛选历史记录。

历史记录支持导出。导出格式包括 JSON（完整数据）和 CSV（摘要数据）两种。导出的数据可用于离线分析和模型训练。

### 5.5 事件系统

#### 5.5.1 事件类型定义

系统定义了以下事件类型。step:start 在步骤开始时触发，包含步骤编号和描述。step:end 在步骤结束时触发，包含步骤结果和耗时。fix:start 在修复开始时触发，包含失败信息和上下文。fix:end 在修复结束时触发，包含修复结果和尝试次数。agent:error 在发生错误时触发，包含错误信息和堆栈。agent:complete 在测试完成时触发，包含汇总统计。

事件消息采用标准格式，包含 eventType（事件类型）、timestamp（事件时间）和 payload（事件数据）三个字段。

#### 5.5.2 事件监听器

事件监听器通过 on 方法注册。方法接收事件类型和回调函数作为参数。回调函数接收事件消息作为参数，支持异步处理。

系统支持通配符监听。使用 * 作为事件类型可以监听所有事件。使用 namespace:* 可以监听命名空间下的所有事件。

监听器支持优先级设置。优先级高的监听器先执行，多个同优先级监听器按注册顺序执行。优先级使用数字表示，默认优先级为 0。

## 六、技能模块功能规格

### 6.1 模块概述

技能模块提供可组合的业务能力抽象，是对底层操作的封装。模块降低了测试用例编写的复杂度，使非技术人员也能编写测试用例。

模块采用注册机制管理技能，支持运行时动态加载和卸载。模块内置 HarmonyOS 平台技能集，也可扩展支持其他平台。

### 6.2 技能基类

#### 6.2.1 Skill 基类

Skill 基类是所有技能的基础。基类包含 metadata（元数据）和 execute（执行方法）两个核心成员。

元数据包含 name（技能名称）、description（描述）、parameters（参数定义）和 examples（使用示例）。参数定义使用 JSON Schema 格式，支持类型验证和默认值设置。

执行方法接收技能参数和执行上下文作为输入，返回技能执行结果。结果包含 success（是否成功）、output（输出数据）和 message（描述信息）。

#### 6.2.2 SkillResult 结果结构

SkillResult 是技能执行结果的统一格式。结果包含 success、output、message 和 metadata 四个属性。success 表示技能是否成功完成。output 是技能产生的数据输出，类型由具体技能决定。message 是结果描述信息，便于人工阅读。metadata 包含执行过程的附加信息如耗时、调用次数等。

### 6.3 HarmonyOS 技能集

#### 6.3.1 登录技能

LoginSkill 封装了登录操作的标准流程。技能接收 username、password 和 loginButton 三个参数。技能执行步骤包括：点击用户名输入框并输入用户名；点击密码输入框并输入密码；点击登录按钮；等待登录完成并验证结果。

技能支持多种登录方式配置。通过 loginType 参数指定登录类型，包括账号密码登录、手机验证码登录和第三方授权登录。每种类型有对应的参数要求和验证逻辑。

技能提供失败重试机制。首次登录失败时，自动等待后重试。重试次数和间隔可配置。多次失败后技能返回失败结果。

#### 6.3.2 表单填写技能

FormFillSkill 封装了表单填写的操作序列。技能接收 formData 参数字典，键为字段名称，值为字段值。技能自动识别表单字段类型，包括文本输入、下拉选择、复选框和单选框。

技能支持智能匹配字段。通过 fieldMapping 参数指定字段名称到元素定位器的映射。映射支持模糊匹配，技能自动找到最匹配的字段进行填写。

技能提供进度回调。填写每个字段后触发回调，报告当前进度。便于在大表单填写时显示进度条。

## 七、知识库模块功能规格

### 7.1 模块概述

知识库模块提供了系统化的知识存储和检索能力。模块存储修复经验、异常模式和领域知识，供系统运行时查询和使用。

模块采用文件存储作为持久化方案，使用 JSON 格式存储知识条目。模块支持全文搜索和向量检索，为不同检索场景提供合适的检索方式。

### 7.2 知识存储子模块

#### 7.2.1 KnowledgeStore 主类

KnowledgeStore 是知识存储的统一入口。主类提供 add（添加知识）、get（获取知识）、update（更新知识）和 delete（删除知识）四个基础方法。

知识条目使用 KnowledgeEntry 结构存储，包含 id（唯一标识）、type（知识类型）、content（内容）、tags（标签）、metadata（元数据）和 version（版本）六个属性。

知识存储使用写时复制策略确保数据一致性。更新操作先写入新版本，待确认成功后才删除旧版本。

#### 7.2.2 向量索引

知识库支持向量索引加速检索。使用嵌入模型将知识内容转换为向量表示。向量存储在向量数据库中，支持高效的相似度搜索。

向量索引在知识添加时自动更新。批量导入知识时，先收集所有知识的向量，再批量写入索引。

索引支持增量更新和全量重建。增量更新只处理新增和变更的知识条目。全量重建重新计算所有知识的向量，用于索引损坏或模型更新场景。

### 7.3 知识检索子模块

#### 7.3.1 SemanticRetriever 语义检索

SemanticRetriever 实现基于语义的检索能力。检索接收自然语言查询，将查询转换为向量表示，在向量索引中搜索最相似的知识条目。

检索结果按相似度排序，支持自定义返回数量。检索支持过滤器，可以按知识类型、标签或时间范围过滤。

检索支持结果扩展。对于相似但不完全匹配的结果，检索自动提取相关内容作为补充显示。

#### 7.3.2 HybridRetriever 混合检索

HybridRetriever 结合语义检索和关键词检索的优点。检索时同时执行两种检索，然后将结果合并排序。

合并算法考虑两种检索结果的相关性评分。语义相关性和关键词相关性按配置权重加权求和。默认权重为语义 0.7、关键词 0.3。

混合检索对短查询和长查询有不同的优化策略。短查询优先语义检索，长查询优先关键词检索。
