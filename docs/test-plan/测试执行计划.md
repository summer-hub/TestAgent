# AI Test Agent TypeScript 重构项目测试计划

## 一、测试计划概述

### 1.1 文档目的

本文档作为 AI Test Agent TypeScript 重构项目的测试执行计划，详细规划测试活动的具体执行细节，包括测试用例设计、测试场景定义、测试数据准备、测试执行安排和测试结果评估。本文档是测试方案文档的具体化，为测试团队提供可直接执行的测试指导。

本文档与测试方案文档形成配套，测试方案定义测试的策略和方法，本文档定义测试的具体执行。测试团队应同时参考两份文档理解和执行测试工作。

### 1.2 测试范围总结

本测试计划覆盖系统的全部功能模块，测试用例总数预估为 850 个。驱动模块测试用例 280 个，覆盖设备连接、元素定位、手势操作、断言验证等功能点。自愈模块测试用例 150 个，覆盖故障诊断、修复决策、修复执行等场景。MCP 模块测试用例 120 个，覆盖协议实现、工具管理、服务调用等功能点。Agent 模块测试用例 200 个，覆盖 ReAct 循环、LLM 集成、测试执行等功能点。技能和知识库模块测试用例 100 个，覆盖技能执行、知识存储检索等功能点。

### 1.3 测试前提条件

测试执行的准备工作应在测试开始前完成，包括测试环境部署完成，所有依赖服务正常运行；测试数据准备完成，所有测试所需的账号、设备和数据已就绪；测试工具配置完成，测试框架运行正常且能够执行测试用例；测试人员培训完成，所有参与测试的人员熟悉测试流程和工具使用；代码冻结完成，待测代码已合并到测试分支且无重大变更。

## 二、测试用例设计

### 2.1 驱动模块测试用例

#### 2.1.1 设备连接测试用例

设备连接测试用例共 25 个，覆盖连接管理的各个功能点。正向测试用例包括：成功建立 TCP 连接并验证连接状态为已连接；连接成功后发送心跳并验证保活机制正常工作；断开连接后验证状态变为已断开且资源已释放；重新连接后验证会话恢复正确。异常测试用例包括：连接目标不存在时验证返回正确的错误信息；连接超时时验证抛出超时异常且不阻塞；连接过程中网络中断时验证自动重连机制；设备序列号无效时验证返回设备未找到错误。

#### 2.1.2 元素定位测试用例

元素定位测试用例共 120 个，覆盖五种定位策略的各种场景。ID 选择器测试用例 30 个，包括：通过存在的 resource-id 精确定位元素并验证返回正确的元素信息；通过存在的 resource-id 部分匹配定位元素；通过不存在的 resource-id 定位时验证返回空结果；resource-id 为空时验证返回参数错误。文本选择器测试用例 35 个，包括：精确匹配屏幕上的文字找到对应元素；模糊匹配部分文字找到包含该文字的元素；使用正则表达式匹配复杂文字模式；使用通配符匹配多种文字模式；匹配多个相同文字时验证返回第一个匹配元素。XPath 选择器测试用例 30 个，包括：使用 child 轴定位子元素；使用 descendant 轴定位后代元素；使用属性匹配定位元素；使用 position 函数定位特定位置的元素；使用复杂的复合条件定位元素。视觉选择器测试用例 15 个，包括：使用模板图片匹配屏幕上的目标区域；多模板同时匹配取最高相似度结果；相似度低于阈值时验证返回空结果；模板图片不存在时验证返回错误。

#### 2.1.3 手势操作测试用例

手势操作测试用例共 80 个，覆盖所有手势类型和参数组合。点击操作测试用例 25 个，包括：单击指定坐标位置并验证点击生效；双击指定位置并验证双击事件触发；长按指定位置并验证长按时间正确；点击存在元素的中心点并验证元素被点击；点击不存在的坐标时验证返回错误。手势操作测试用例 30 个，包括：向上滑动指定距离并验证滑动轨迹正确；向下滑动并验证方向和距离正确；向左滑动并验证无偏差；向右滑动并验证无偏差；对角线滑动并验证轨迹平滑。拖拽操作测试用例 15 个，包括：从起点拖拽到终点并验证终点位置正确；拖拽过程中手指离开屏幕验证拖拽中断；拖拽目标不存在时验证返回错误。缩放操作测试用例 10 个，包括：双指放大操作并验证放大效果；双指缩小操作并验证缩小效果；缩放目标不存在时验证返回错误。

#### 2.1.4 断言验证测试用例

断言验证测试用例共 55 个，覆盖软断言和硬断言的各种场景。文本存在断言测试用例 20 个，包括：屏幕上存在目标文本时验证断言通过；屏幕上不存在目标文本时验证断言失败；存在多个目标文本时验证返回第一个匹配；使用模糊匹配时验证阈值正确应用。组件可见断言测试用例 20 个，包括：元素可见时验证断言通过；元素不可见时验证断言失败；元素不存在时验证返回元素未找到错误；元素被遮挡时验证返回可见性检查失败。属性匹配断言测试用例 15 个，包括：属性值匹配时验证断言通过；属性值不匹配时验证断言失败；属性不存在时验证返回属性错误；多个属性同时匹配时验证逻辑正确。

### 2.2 自愈模块测试用例

#### 2.2.1 故障诊断测试用例

故障诊断测试用例共 60 个，覆盖 10 种失败类型的诊断。正向诊断测试用例包括：ElementNotFound 类型错误验证正确分类为 ELEMENT_NOT_FOUND；AuthExpired 类型错误验证正确分类为 AUTH_EXPIRED；AssertionTooStrict 类型错误验证正确分类为 ASSERTION_TOO_STRICT；Timeout 类型错误验证正确分类为 TIMEOUT；NetworkError 类型错误验证正确分类为 NETWORK_ERROR。每种类型额外测试边界情况如错误消息格式异常、错误类型混合等。

诊断置信度测试用例包括：规则完全匹配时验证置信度达到 90% 以上；规则部分匹配时验证置信度在 60% 到 85% 之间；无匹配规则时验证分类为 UNKNOWN_ERROR；多个规则匹配时验证选择权重最高的规则。

证据收集测试用例包括：截图证据在诊断前正确捕获；UI 树证据包含完整的元素信息；错误日志证据正确提取相关记录；证据按时间戳正确排序。

#### 2.2.2 修复决策测试用例

修复决策测试用例共 45 个，覆盖决策引擎的各个功能点。策略选择测试用例包括：ELEMENT_NOT_FOUND 类型建议选择器修复策略；AUTH_EXPIRED 类型建议认证修复策略；ASSERTION_TOO_STRICT 类型建议断言修复策略；DATA_POLLUTED 类型建议数据修复策略；TIMEOUT 类型建议等待修复策略。

优先级排序测试用例包括：历史成功率高的策略排在前面；置信度高的策略优先选择；尝试次数少的策略优先选择；同时考虑三个因素时权重计算正确。

重试控制测试用例包括：尝试次数达到上限后停止继续尝试；成功修复后重置该用例的尝试次数；不同用例的尝试次数独立计数。

#### 2.2.3 修复执行测试用例

修复执行测试用例共 45 个，覆盖修复流程的各个阶段。选择器修复测试用例 20 个，包括：文本定位失败时自动切换到 ID 定位；ID 定位失败时自动切换到 XPath 定位；XPath 定位失败时自动切换到视觉定位；修复后验证元素成功找到。认证修复测试用例 10 个，包括：检测到 token 过期时自动重新登录；登录凭证有效时验证登录成功；token 刷新成功时验证新 token 生效；多次认证失败后验证停止重试。断言修复测试用例 8 个，包括：断言失败时自动放宽匹配条件；放宽后断言通过验证修复成功；修复失败时验证返回修复失败状态。等待修复测试用例 7 个，包括：超时时自动增加等待时间；等待时间增加到最大值后验证停止；增加等待后操作成功验证修复成功。

## 三、测试场景设计

### 3.1 核心功能测试场景

#### 3.1.1 端到端测试场景

端到端测试场景验证完整业务流程的运行，覆盖从测试用例提交到结果获取的全链路。

场景一：完整测试执行流程。准备一个包含 5 个步骤的测试用例，提交给 Agent 执行。验证 Agent 能够依次完成每个步骤的思考、行动和观察过程。验证所有步骤通过时最终结果为成功。验证执行完成后生成包含详细步骤信息的执行报告。

场景二：失败自动修复流程。准备一个故意设计为失败的测试用例。验证测试执行到失败步骤时系统自动进入修复流程。验证诊断器正确识别失败原因。验证决策引擎选择了合适的修复策略。验证修复后重新执行步骤。验证修复成功时最终结果为通过。

场景三：多步骤复杂场景。准备一个包含分支和循环的测试计划。验证 Agent 能够正确处理分支选择。验证循环次数正确控制。验证执行过程中的状态正确维护。验证最终统计结果正确。

#### 3.1.2 并发测试场景

并发测试场景验证系统在并发负载下的稳定性。

场景一：多 Agent 并发执行。启动 5 个 Agent 实例同时执行不同的测试用例。验证每个 Agent 的执行结果独立正确。验证系统资源使用在合理范围内。验证无并发相关的数据竞争问题。

场景二：高频工具调用。单个测试用例中设计高频的工具调用序列。验证每个调用都能正确执行和返回。验证调用结果的顺序正确。验证无内存泄漏或资源耗尽问题。

### 3.2 异常处理测试场景

#### 3.2.1 网络异常场景

网络异常场景验证系统在网络问题下的容错能力。

场景一：设备连接中断。执行测试过程中断开设备连接。验证系统检测到连接中断。验证自动重连机制正常工作。验证重连后能够恢复执行。验证执行结果正确记录中断和恢复过程。

场景二：LLM API 不可用。执行测试过程中模拟 LLM API 返回 503 错误。验证系统正确处理服务不可用错误。验证错误信息清晰明确。验证提供重试或降级选项。

场景三：部分网络抖动。执行测试过程中模拟间歇性的网络延迟。验证系统能够容忍一定程度的延迟。验证执行时间合理增加但结果正确。验证无超时导致的失败误判。

#### 3.2.2 系统异常场景

系统异常场景验证系统在意外情况下的恢复能力。

场景一：进程异常终止。执行测试过程中强制终止测试进程。验证关键状态正确保存。验证可以从断点恢复执行。验证恢复后执行结果与中断前一致。

场景二：内存压力。执行大量测试用例后检查内存使用。验证内存增长在合理范围内。验证触发垃圾回收后内存释放。验证无内存泄漏导致的性能退化。

场景三：磁盘空间不足。执行过程中填充磁盘使空间不足。验证系统正确检测到空间不足。验证写入操作失败但不影响其他功能。验证恢复空间后功能恢复正常。

### 3.3 性能测试场景

#### 3.3.1 响应时间测试场景

响应时间测试场景验证系统满足性能需求。

场景一：单步执行响应时间。执行 100 个单步骤测试用例，记录每个步骤的响应时间。验证平均响应时间低于 500 毫秒。验证 95 分位响应时间低于 1 秒。验证最差情况响应时间低于 5 秒。

场景二：批量执行响应时间。执行包含 100 个测试用例的测试计划。验证总执行时间低于 10 分钟。验证平均每个用例执行时间合理。验证批量执行吞吐量达标。

#### 3.3.2 稳定性测试场景

稳定性测试场景验证系统长时间运行的可靠性。

场景一：24 小时持续运行。持续执行测试用例 24 小时，模拟生产环境负载。验证系统无崩溃或异常退出。验证性能指标无明显退化。验证资源使用稳定无泄漏。

场景二：峰值负载测试。模拟 3 倍正常负载的测试场景。验证系统能够处理峰值而不崩溃。验证性能下降在可接受范围内。验证负载降低后系统恢复正常。

## 四、测试数据准备

### 4.1 测试数据需求

测试数据分为五类：设备数据、用户数据、测试用例数据、预期结果数据和配置数据。

设备数据包括 HarmonyOS 设备的序列号、IP 地址、屏幕分辨率、操作系统版本等信息。测试环境应配置至少 2 台可用设备，主备切换测试需要至少 3 台设备。

用户数据包括测试账号的用户名、密码和权限信息。测试账号应覆盖不同权限级别，包括管理员、普通用户和只读用户。测试完成后应清理测试账号创建的数据。

测试用例数据包括测试用例的 JSON 定义和测试计划配置。测试数据应覆盖正向场景和异常场景的边界条件。

预期结果数据包括每个测试用例的预期执行结果。预期结果应包含步骤级别的详细预期，而非仅整体预期。

配置数据包括不同测试环境下的配置文件。配置数据应覆盖默认配置、特殊配置和异常配置场景。

### 4.2 测试数据准备流程

测试数据准备分为准备阶段、执行阶段和清理阶段。

准备阶段在测试执行前完成。创建测试所需的设备账号和设备配置。准备测试用例文件和测试数据文件。初始化测试数据库和测试文件系统。验证所有测试数据就绪且可用。

执行阶段在测试运行期间维护。监控测试数据的消耗情况。及时补充被消耗的数据。记录测试数据的消耗和变化情况。

清理阶段在测试执行后完成。清理测试创建的数据文件。恢复测试修改的配置文件。重置测试使用的设备状态。归档测试过程产生的日志和报告。

### 4.3 测试数据管理

测试数据采用集中管理的策略。测试数据存储在统一的测试数据仓库中，所有测试共享同一数据源。测试数据使用版本管理，变更需要评审和记录。测试数据有明确的责任人，负责数据维护和变更。

## 五、测试执行安排

### 5.1 执行阶段划分

测试执行分为四个阶段：冒烟测试、功能测试、集成测试和验收测试。

冒烟测试在每个迭代结束后执行，执行时间 1 小时，覆盖核心功能的快速验证。功能测试在功能开发完成后执行，执行时间 1-2 天，覆盖所有功能需求的验证。集成测试在系统集成完成后执行，执行时间 2-3 天，覆盖模块间集成的验证。验收测试在发布前执行，执行时间 1 周，覆盖非功能需求和端到端场景的验证。

### 5.2 执行日程安排

测试执行日程安排如下：

第 1 周执行冒烟测试，执行 Phase 2 开发的驱动模块和 MCP 模块的冒烟用例。发现缺陷立即反馈开发修复。

第 2 周继续执行冒烟测试，执行 Phase 3 开发的 Agent 模块的冒烟用例。收集所有模块的冒烟测试结果。

第 3-4 周执行功能测试，按模块分组执行完整的功能测试用例。每日汇总测试结果和缺陷。

第 5-6 周执行集成测试，执行模块间集成的测试用例和接口测试。验证端到端的数据流正确性。

第 7 周执行性能测试，执行响应时间和稳定性测试场景。收集性能数据并分析。

第 8 周执行验收测试，执行完整的端到端测试场景和异常场景。邀请用户代表参与验收。

### 5.3 执行人员安排

测试执行人员安排如下：

测试开发工程师负责所有自动化测试的执行和结果分析。在每个执行阶段全程参与。每日向项目经理汇报测试进展。

开发工程师负责手工测试用例的执行和缺陷验证。在功能测试阶段按模块分配负责的测试用例。发现缺陷后负责修复和重新验证。

架构师负责测试结果的评审和验收决策。在验收测试阶段参与关键的验收判断。提供技术指导解决测试中发现的问题。

## 六、测试结果评估

### 6.1 通过标准

功能测试的通过标准为：测试用例执行率 100%，即所有设计的测试用例都已执行；测试用例通过率 95% 以上，即通过率需达到要求；无高优先级或严重缺陷遗留，即 P0 和 P1 缺陷必须全部修复；无阻塞性问题，即无阻止测试继续执行的阻塞问题。

性能测试的通过标准为：响应时间满足需求规格，即所有性能指标达标；稳定性测试无异常，即长时间运行无崩溃或严重退化；资源使用合理，即 CPU、内存和磁盘使用在预期范围内。

### 6.2 评估指标

测试结果评估使用以下指标：

测试执行率是已执行测试用例数除以设计的测试用例总数，目标 100%。测试通过率是通过的测试用例数除以执行的测试用例总数，目标 95% 以上。缺陷密度是发现的缺陷数除以代码行数，用于评估代码质量。缺陷修复率是已修复的缺陷数除以发现的缺陷总数，目标 100%。测试覆盖率是已覆盖的代码行数除以总代码行数，目标 80% 以上。

### 6.3 测试报告

测试完成后编制测试报告，包含以下内容：

测试执行概况包括测试时间、测试人员、测试环境等基本信息。测试执行统计包括执行的用例数、通过率、失败率等数据。缺陷统计包括发现的缺陷数、按严重程度分布、按模块分布等。性能评估包括响应时间测试结果、稳定性测试结果。遗留问题包括无法修复或暂不修复的问题列表。测试结论包括是否通过验收的建议。

## 七、缺陷跟踪

### 7.1 缺陷优先级定义

缺陷优先级分为四个等级：

P0 优先级是最高优先级，指系统核心功能不可用或存在数据丢失风险，必须立即修复。P1 优先级是高优先级，指系统重要功能无法正常工作，必须在 24 小时内修复。P2 优先级是中优先级，指系统存在缺陷但有替代方案，需要在 1 周内修复。P3 优先级是低优先级，指系统存在不影响功能的瑕疵，可以在后续版本中修复。

### 7.2 缺陷状态流转

缺陷状态包括新建、已确认、已分配、修复中、已修复、验证中、已关闭七种状态。

新建状态是测试人员提交缺陷报告后的初始状态。已确认状态是项目经理评审并确认缺陷有效后的状态。已分配状态是缺陷分配给开发人员处理的状态。修复中是开发人员开始修复代码的状态。已修复是代码修复完成等待验证的状态。验证中是测试人员验证修复结果的状态。已关闭是验证通过缺陷正式关闭的状态。

### 7.3 缺陷报告要求

缺陷报告必须包含以下信息：缺陷标题简洁明确，准确描述问题；缺陷描述详细完整，包含问题的背景和影响；复现步骤清晰可操作，能够指导开发人员复现问题；环境信息准确记录，包括操作系统版本、Node.js 版本等；缺陷截图或日志辅助理解问题；严重程度和优先级明确定义。

## 八、测试退出标准

### 8.1 测试完成条件

测试阶段完成的退出标准包括：所有计划的测试用例都已执行完成；所有 P0 和 P1 缺陷都已修复并验证通过；测试执行率达到 100%；测试用例通过率达到 95% 以上；测试覆盖率满足要求；测试报告已完成编制。

### 8.2 发布条件

系统发布的测试前提条件包括：验收测试全部通过；所有已知缺陷的状态都为已关闭或已确认推迟；性能测试结果满足需求指标；安全性测试未发现高风险漏洞；文档完备且通过评审。
